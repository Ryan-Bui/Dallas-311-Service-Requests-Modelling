This document provides a structured, comprehensive knowledge representation of the City of Dallas 311 Service
Request dataset. It is designed for ingestion into a knowledge graph, enabling entity resolution, relationship traversal,
semantic querying, and downstream analytics. All entity types, attributes, relationships, business rules, statistical
distributions, and temporal patterns are documented in detail.
1. Dataset Overview
Dataset Name
City of Dallas 311 Service Requests
Source
City of Dallas Open Data Portal (311 system)
Record Count
3,100 service request records (sample)
Field Count
16 attributes per record
Date Range
January 2020 ΓÇô January 2026
Geographic Scope
City of Dallas, Texas; 14 City Council Districts
Primary Key
Service Request Number (e.g. 22-00056736); also Unique Key (Salesforce ID)
Coordinate System
WGS84 decimal degrees (latitude, longitude)
Primary Use Cases
Service delivery monitoring, workload analytics, SLA compliance, geographic
analysis, predictive routing
The 311 system is the City of Dallas's non-emergency municipal service line. Citizens, businesses, and city staff
submit requests for a wide range of services ΓÇö from pothole repair and code violations to animal control and
sanitation. Each request is assigned to a city department, given an estimated response time, and tracked through to
resolution.

6. Business Rules & Domain Knowledge
6.1 Service Request Lifecycle
A service request follows this lifecycle in the 311 system:
New: Request received and logged; not yet assigned or actioned.
In Progress: Assigned to a department/officer; work underway.
Closed: Request fully resolved. Outcome recorded.
Closed (Transferred): Request was routed to another department or system. May appear without Outcome.
Closed (Duplicate): Identified as a duplicate of an existing request. Merged with parent.
6.2 Priority Tier Definitions
Emergency: Immediate life/safety risk. Fastest response. Triggers dispatch protocol. Example: gas leak, downed
power line.
Dispatch: Requires field dispatch within hours. Includes animal aggressive behavior, signal outages, road
hazards.
Priority: Elevated urgency above Standard but below Dispatch. Accelerated SLA.
MCC: Multi-Channel Coordination ΓÇö requests requiring coordination across multiple city departments.
High: Elevated priority flag; rare in dataset (3 records). May be system or operator-assigned.
Standard: Default priority for routine service requests. Governed by ERT in business or calendar days.

6.4 Intake Channel Classification
Phone: Citizen or business calls 3-1-1. Logged by call center agent. Highest human-interaction intake.
API: Programmatic submission via Salesforce or city systems. Often bulk or automated (e.g., inspection systems).
Proactive: City staff identifies issue during patrol, inspection, or monitoring without citizen complaint.
Email: Submission via email to 311 address. Lower volume; may carry attachments.
Internal: Originating from within a city department. Used for inter-departmental transfers or self-reported issues.
311-App: Mobile app submission by citizen. Includes GPS location from device.
Web: Web form submission via City of Dallas website.
6.5 SLA and Response Time Rules
 ERT is expressed as Business Days (MonΓÇôFri, excluding holidays) or Calendar Days (all days) or Hours.
 The Due Date is computed as: Created Date + ERT value. If ERT is null, no Due Date is generated.
 Requests are considered 'on-time' if Closed Date <= Due Date.
 Dispatch priority requests typically have ERTs of 4 Hours or 1 Business Day.
 Emergency priority requests may have no formal ERT but are expected to be actioned immediately.
 MCC requests may span multiple departments and have extended ERTs (up to 120 Business Days observed).
 110 and 120 Business Day ERTs are associated with long-duration compliance or construction monitoring
cases.
6.6 Geographic Rules
 All requests within Dallas city limits. Coordinates are WGS84 decimal degrees.
 Dallas bounding box: Latitude 32.62ΓÇô33.02 N, Longitude -97.00 to -96.55 W.
 One record has dual-district assignment (14,9) indicating a boundary edge case.
 Intersection-based addresses (e.g., MCKINNEY AVE & FAIRMOUNT ST) are used when no specific property is
involved (signal repairs, road work).
 ZIP codes in addresses range across multiple Dallas ZIP code areas.
 The Lat_Long Location field requires parsing: extract numeric values from tuple string format '(lat,long)'.
6.7 Duplicate & Transfer Rules
 Closed (Duplicate) requests (73 records, 2.4%) are de-duplicated against an existing parent request. No
independent resolution outcome is recorded.
 Closed (Transferred) requests (156, 5.0%) indicate the request was moved to another department, system, or
jurisdiction. Outcome may be blank.
 Duplicate detection relies on address match + service type + proximity in time.
 Transferred requests may continue under a different Service Request Number in the receiving department.

8. Recommended Query Patterns
Representative queries for traversing the knowledge graph once ingested. Patterns shown in pseudo-Cypher (Neo4j
style).
Find all open service requests in District 4
MATCH (sr:ServiceRequest)-[:IN_DISTRICT]->(d:CouncilDistrict {number: 4}) WHERE sr.status IN
['New', 'In Progress'] RETURN sr.requestNumber, sr.serviceType, sr.createdDate, sr.priority
List all service types handled by Code Compliance
MATCH (dept:Department {name: 'Code Compliance'})<-[:MANAGED_BY]-(srt:ServiceRequestType) RETURN
DISTINCT srt.name ORDER BY srt.name
Average resolution time by department for Dispatch priority
MATCH (sr:ServiceRequest)-[:ROUTED_TO]->(d:Department) WHERE sr.priority = 'Dispatch' AND
sr.daysToClose IS NOT NULL RETURN d.name, avg(sr.daysToClose) AS avgDays, count(sr) AS total
ORDER BY avgDays
Find clusters of Code Concerns in the same block
MATCH (sr1:ServiceRequest)-[:OF_TYPE]->(t:ServiceRequestType {name: 'Code Concern - CCS'}) MATCH
(sr1)-[:SUBMITTED_AT]->(a1:Address) MATCH (sr2)-[:SUBMITTED_AT]->(a2:Address) WHERE a1.zipCode =
a2.zipCode AND sr1 <> sr2 RETURN a1.zipCode, count(sr1) AS count ORDER BY count DESC
Requests overdue (past due date, still open)
MATCH (sr:ServiceRequest) WHERE sr.status IN ['New','In Progress'] AND sr.dueDate IS NOT NULL AND
sr.dueDate < datetime() RETURN sr.requestNumber, sr.department, sr.dueDate, sr.priority ORDER BY
sr.dueDate
Top outcomes for Animal Services requests
MATCH (sr:ServiceRequest)-[:ROUTED_TO]->(d:Department {name: 'Dallas Animal Services'}) MATCH
(sr)-[:RESOLVED_WITH]->(o:Outcome) RETURN o.name, count(sr) AS cnt ORDER BY cnt DESC

9.4 Temporal Graph Considerations
 Store Created Date, Closed Date, and Update Date as datetime properties on the ServiceRequest node.
 Create a computed edge weight property on ROUTED_TO equal to days_to_close for path-based SLA queries.
 Consider a separate temporal index or time-tree (Year ΓåÆ Month ΓåÆ Day) for efficient date-range queries.
 Versioned status tracking: if status history is available in full 311 export, model as Status event nodes with
